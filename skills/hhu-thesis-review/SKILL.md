# 河海大学博士论文审稿（五部分）

## 这个 skill 做什么

对河海大学博士学位论文的五个固定部分给出可操作的修改意见：**前言、摘要、问题的提出（绪论 1.1）、结论（结论与展望）、参考文献**。审查对象是**写作质量与著录格式**。

审查分两层结合：

- **读者代入判断**：代入一位熟悉领域、但没读过本文的同行评审，把每一块读一遍，老实判断"我读完到底懂没懂、抓没抓住、感觉到没感觉到"。最典型的用处是创新点——判断作者**有没有把研究的新意表述清楚**（读完能不能一句话说清新在哪），并指出常见病：把"研究内容的复述"当成"创新点凝练"、动词不承载新意、真正的创新核心被一句话写薄。
- **条目清单检查**：每块还有一套具体检查项（背景漏斗、关键词、参考文献著录格式等），查"缺什么"。

**一条重要边界**：本 skill 审的是创新点**有没有写清楚、新意读不读得出来**（这是写作表达问题），**不是**评判这个创新点在学科上**够不够格、价值大不大**（那是学科判断，依赖具体领域知识，超出范围）。同样，不评判方法选得对不对、论证严不严密。简单说：我们帮作者把"他自己认定的贡献"表述到位，不替他判断这个贡献够不够分量。

审稿规则来自对真实样本的归纳，但更要紧的是上面那层读者代入判断——它不依赖预先列好的规则，而是当场代入读者去检验"写到位了没有"，正好补上机械清单查不出的问题（要素齐全却读不出新意）。理解规则背后的"为什么"比机械套用更重要：河海博士论文有一套稳定的体例习惯，也有几类反复出现的通病，下面的清单就是针对这些习惯和通病设计的。

## 河海体例的一个关键前提

在河海博士论文里，**"前言"和"摘要"是两个独立部分，分工不同**：

- **前言（foreword）**：偏研究背景与创新点导向，回答"为什么做、新在哪"。
- **摘要（abstract）**：偏研究结论导向，回答"做了什么、得到什么结果"。

两者都习惯用阿拉伯数字分条列贡献，因此很容易写得高度雷同。审前言和审摘要时要分别按各自的定位来要求，并主动检查两者是否过度重复（见下方红线）。

## 怎么用

1. **判断要审哪几块**。用户可能给整篇，也可能只丢一段说"帮我看看摘要""参考文献格式对不对"。**只审用户要求的那一块（或几块）**，不要强行五块全做。若用户给了整篇但没指定，默认五块都审。
2. **逐块审查**。每一块都有专门的检查清单，见 `references/section-checklists.md`（前言、摘要、问题的提出、结论）和 `references/gb-t-7714.md`（参考文献，按 GB/T 7714—2015 顺序编码制）。审某一块前，先读对应清单再动手。**每块都先做清单里的"第 0 条·读者代入判断"再走条目检查**——代入一位熟悉领域但没读过本文的同行，先老实判断"我读完到底懂没懂、抓没抓住、感觉到没感觉到新意"，把这个判断显式写成意见，再用条目清单查缺补漏。这一步是本 skill 的关键：机械的要素清单查得出"缺什么"，查不出"写得到不到位"（典型如创新点要素齐全却读不出新意），必须靠读者代入判断补上。
3. **先查跨部分红线**（见下一节），再进入单块清单。
4. **生成 Word 文档作为最终交付物**（见"输出格式：最终交付一个 Word 文档"）。用户通常一次只审一块（如只对摘要生成逐句表），**不强制整篇五块一次性生成**——审哪块就只为哪块出文档。

## 跨部分红线（最优先，务必先查）

这几条是样本里最典型、危害最大的问题，**置顶检查**：

1. **背景模板三处复用【硬伤】**：同一段研究背景（往往是开头那段"某流域/某领域具有……重要作用"）被几乎逐句搬到前言、摘要、结论三处。这是河海博士论文头号通病。做法：把三处开头段落两两比对，重合度高就标为硬伤，建议结论章删去背景、直奔结论，前言与摘要的背景各自精简且不互相照抄。
   - **单块模式收敛**：当用户**只审某一块**（如只审摘要）时，无法跨块比对，**不要**为此单独起一节、不展开改法。只在该块文档末尾用**一句话脚注**提醒（如"摘要开头属典型背景模板，请自查前言/结论是否也照搬同段"），保持单块聚焦的边界感。只有**整篇审查**时才按上面置顶硬伤的方式完整处理。
2. **引用序号断链【硬伤】**：正文引用序号与文末参考文献列表对不上（断号、跳号、引而未列、列而未引）。详见参考文献清单的交叉校验部分（注意：这是"尽力而为"项，见该节说明）。
3. **问题—内容—结论闭环断裂【建议】**：问题的提出里点出的科学问题，应能与研究内容、主要结论一一对应。断链处要标出。

## 严重度分级

每条意见都要打一个标签，让作者知道轻重缓急：

- **【硬伤】**：硬性缺陷或会被答辩/盲审直接挑出的问题，必须改。例如背景模板三处复用、引用序号断链、参考文献缺著者/缺年份、结论纯属过程复述。
- **【建议】**：明显影响质量、强烈建议改，但不至于"硬错"。例如关键词漏核心词、创新点缺"结果"、展望空泛。
- **【可选】**：润色级、锦上添花，改不改都行。例如个别长句拆分、措辞优化。

## 输出格式：最终交付一个 Word 文档

**最终交付物是一个 Word（.docx）文档**，不是只在对话里给散文意见。对话里可以先给**一段简短总览**（命中的主要问题、优先级），但要让用户能下载 .docx、逐行落实修改。

### 文档结构（两大区）

1. **顶部"总体与跨段问题"区**：放**无法定位到单一句子**的结构性/跨段问题——背景模板三处复用（红线）、创新点与摘要结论重复、关键词漏核心词、结论缺整体升华、问题—内容—结论闭环断裂等。每条带严重度标签（硬伤/建议/可选）。

2. **逐句修改表**：四列——

   | 原文定位 | 问题 | 修改后 | 修改理由 |
   |---|---|---|---|
   | 摘出有问题的那一句（≤一句话） | 问题描述（含【严重度】标签） | 改写后的完整句子 | 为什么改 + 该怎么改的动作 |

   - **只列有问题的句子**，不是把用户原文逐句全列；没问题的句子不进表。
   - **"修改后"给完整改写句**；若涉及自拟数字，用占位符（见下"数字防呆"）。
   - **"修改理由"把"该怎么改"的动作一并说清**，让用户照着就能改。

### 怎么生成这个文档

用 `scripts/build_review_docx.py` 生成（基于 python-docx，已处理中文 eastAsia 字体与表格边框等坑，**不要从零写 docx 代码**）。流程：

1. 按清单审完后，把结果组织成一个 JSON（结构见 `references/docx-output.md`）。
2. 运行：`python scripts/build_review_docx.py <spec.json> <输出路径.docx>`。
3. 把 .docx 提供给用户下载。

具体字段、示例 JSON、字体实现要点见 `references/docx-output.md`，生成前必读。

### 三条贯穿原则（写每条意见都遵守）

- **可定位**：指明是哪一句、哪一条创新点/结论/参考文献，不要泛泛说"摘要不够好"。逐句表的"原文定位"列就是为此而设。
- **解释"为什么"再给"怎么改"**：作者要理解问题本质才能举一反三，不要只下判断。
- **示范/改写句的合规与数字防呆**（重要）：
  - 若举**反例**，可摘真实"差句"，但必须**极短（≤25字）且脱去任何作者、单位、课题标识**，不可整段照搬原文当语料。
  - **正例 / "修改后" 一律由你按规则现场生成**，不摘任何真实论文里的好句。
  - **数字防呆**：自拟示范句里的定量结果**一律用明显占位符**（如 `X%`、`Δ天`、`相关系数 X`、`约 X m/s`），**不要写看起来像真实测值的具体数字**（如"35%""4–6 天"）。目的是防止用户把示范数字误抄进论文。文档生成时会在显著位置统一提示"表中数字为示意，请替换为你的实测值"。

## 边界

- 只做五块的写作与格式审查。注意区分两件事：**审"创新点有没有表述清楚、新意读不读得出来"是本 skill 的核心工作**（见读者代入判断）；但用户若要你**评判"这个创新点在学科上够不够格、价值大不大""论证逻辑是否成立""方法是否合适"**，这些是学科判断，超出本 skill 范围，说明清楚，可改为就写作表达提意见。一句话：帮作者把贡献说清楚，不替作者判断贡献够不够分量。
- 看不到原始 PDF 排版时（例如只拿到抽取文本），参考文献的交叉校验只能尽力而为，要如实说明，见 `references/gb-t-7714.md`。
