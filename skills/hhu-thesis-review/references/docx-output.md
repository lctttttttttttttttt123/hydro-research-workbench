# 生成审稿 Word 文档

最终交付物是一个 .docx。审完后，把结果组织成一个 JSON spec，交给 `scripts/build_review_docx.py` 生成。**不要自己从零写 python-docx 代码**——脚本已处理中文 eastAsia 字体、表格边框、字号、数字防呆提示等坑。

## 生成步骤

1. 把审稿结果写成 `spec.json`（结构见下）。
2. 运行：
   ```bash
   python scripts/build_review_docx.py spec.json 审稿意见.docx
   ```
   （需要 python-docx：`pip install python-docx --break-system-packages`）
3. 把生成的 .docx 交给用户下载。

## spec.json 结构

```json
{
  "title": "审稿意见：摘要",
  "scope": "本次审查范围：摘要（共 1 块）",
  "overview": "可选，一段简短总览，放文档开头（也可省略，对话里给即可）。",
  "cross_section_issues": [
    {"severity": "硬伤", "text": "无法定位到单句的结构性/跨段问题描述。"}
  ],
  "sentence_edits": [
    {
      "locate": "原文里有问题的那一句（≤一句话，可适当截断）。",
      "issue": "【硬伤】问题描述，开头放严重度标签【硬伤/建议/可选】。",
      "revised": "改写后的完整句子，数字用占位符 X% / Δ天 / 相关系数 X。",
      "reason": "为什么改 + 该怎么改的动作。"
    }
  ],
  "single_block_footnote": "可选。单块模式下关于背景模板复用等跨段提醒的一句话脚注。"
}
```

### 字段填写规则（对应 SKILL.md 的要求）

- **cross_section_issues（总体与跨段问题区）**：放**无法定位到单句**的问题——背景模板三处复用、创新点与摘要结论重复、关键词漏核心词、结论缺整体升华、问题—内容—结论闭环断裂等。每条必须有 `severity`（硬伤/建议/可选）。整篇审查时，背景模板三处复用放这里并标硬伤；**单块模式下不要把它放这里**，改用 `single_block_footnote` 一句话带过。
- **sentence_edits（逐句修改表）**：
  - **只放有问题的句子**，没问题的不进表。
  - `issue` 开头务必带 `【硬伤】/【建议】/【可选】`（脚本会据此给该列上色）。
  - `revised` 给**完整改写句**；凡定量结果**一律用占位符**（`X%`、`Δ天`、`相关系数 X`、`约 X m/s`），不要写"35%""4–6 天"这类看似真实的数字。脚本顶部会自动加一句"数字为示意，请替换为你的实测值"。
  - 若引用真实"差句"作 `locate`，须 **≤一句话且脱去任何作者/单位/课题标识**。
- **single_block_footnote**：仅单块模式用。例：`摘要开头属典型背景模板，请自查前言/结论是否也照搬同段。`

## 输出形态约定

- 文档不必符合论文正文排版规范，**清晰可读即可**，但**中文字体必须正确**（脚本已设宋体/黑体 eastAsia）。
- 标题黑体居中；"一、总体与跨段问题"用项目符号列出带色严重度标签；"二、逐句修改表"为四列带框表格，表头灰底。
- 对话里可先给一段简短总览，但**务必生成并交付 .docx**，让用户能下载、逐行落实。

## 多块审查

用户一次只审一块时，`title`/`scope` 就写那一块，逐句表只含那块的句子。若用户给整篇要五块全审，可在同一文档里按块分组：可在 `sentence_edits` 的 `locate` 前自行注明所属部分（如"［摘要］……"），或分多次生成多个文档——以单块聚焦、清晰为先。
