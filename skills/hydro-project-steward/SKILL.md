# 可复现项目工作区 · 项目管家

让这套工作台的产出不再是散落的文件,而是一个**可复现、可审计**的项目:每张图、每份成品
都能追溯到"哪份数据、哪个脚本、什么参数、当时为什么这么做"。本技能负责工作区的搭建与
维护;写作/制图/分析由其他专项 skill 完成。

## 何时启动
- **课题开始**:任何会产出数据/图/稿件的多步任务,先建项目工作区,再干活。
- **过程中**:每产出一张图或一份成品,登记归档;每个关键决策或缺口,记日志。
- **交付前**:跑复现自检,确认一切可追溯、待补已清。

## 工具(在 scripts/ 下)
- `init_project.py "<课题名>" [--author 姓名]` —— 生成标准项目目录(data/scripts/figures/
  refs/drafts/outputs/logs/env + README + manifest.json)。
- `register.py register --root <项目> --type figure|output|dataset|draft --path ... --data ...
  --script ... --params ... --title-zh ... --title-en ...` —— 登记产出进 manifest,图会自动
  写出同名 `.meta.json`(可追溯)。
- `register.py pending --root <项目> --note "..."` —— 汇总一条待补/待核。
- `register.py log --root <项目> --note "..."` —— 追加一条工作日志。
- `register.py freeze --root <项目>` —— 锁定复现环境(pip freeze 到 env/)。
- `register.py check --root <项目>` —— 交付前复现完整性自检。

## 标准工作法(务必遵守)
1. **开工先建区**:`init_project.py "课题名"`。把原始数据放进 `data/raw/`,并为每个数据集
   写 `.source.md`(来源、条件、单位、字段)。原始数据**只读不改**。
2. **结果只由脚本产生**:派生数据(`data/processed/`)与图(`figures/`)只能由 `scripts/`
   里的脚本生成,不手改。这样任何结果都能重跑复现。figure-composer 出图后,用
   `register.py register --type figure` 登记并写 meta。
3. **边干边记**:每产出一份成品就 `register`;每个关键决策、参数选择、发现的缺口就 `log`
   或 `pending`。日志是"可审计"的核心——几个月后靠它复原"当时为什么这么做"。
4. **稿件留版**:综述/章节/摘要,以及去AI味前后、审稿前后,各存一版进 `drafts/`,不覆盖。
5. **交付前自检**:`register.py check`。它会检查:图有没有 meta(可追溯)、派生数据有没有
   对应脚本、待补是否清零、环境是否已锁。**有问题先补齐再交付。**

## 与"不编造"铁律的关系
本工作区把全局铁律落地为机制:任何图/数据/结论若非实测或脚本计算得来,不得进 `figures/`
或写入结论,而应 `pending` 记为 `【待补】`;交付前 `check` 会拦下没清零的待补。图没有
meta = 来源不明 = 视为不可信,必须补 meta 或重生成。

## 在整套流程中的位置
它是贯穿全程的"底座",不是某一步:
- 课题开始 → 本技能建区;
- 检索/综述/分析/制图/写作/审稿/去AI味/成文 各步的产出 → 都经本技能登记归档;
- 成文(hhu-thesis-docx)前后 → 登记成品、跑 check、freeze 环境。

## 不做什么
不写正文、不画图、不做数据分析、不检索文献——那些交给对应专项 skill。本技能只管
"工作区、归档、日志、复现自检"。
